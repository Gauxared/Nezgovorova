# Целевая система

## Цель

Система должна организовать работу внешних сотрудников по лидам компании и дать руководителю прозрачную аналитику обработки лидов.

Практический результат:

- CRM-данные из Bitrix24 попадают в нашу БД через API;
- сотрудник получает ежедневный список задач в Telegram;
- сотрудник вечером сообщает прогресс по каждому лиду или задаче;
- ответы из Telegram сохраняются в нашей системе;
- важные результаты уходят обратно в Bitrix24;
- аналитика показывает состояние лидов, нагрузку, просрочки, конверсию и качество обработки.

## Роли

- `admin` - настройка интеграций, пользователей, справочников, прав и системных параметров.
- `manager` - контроль лидов, сотрудников, отчетов и проблемных зон.
- `operator` - внутренний сотрудник, который может корректировать данные и контролировать внешних исполнителей.
- `external_worker` - внешний сотрудник, работающий через Telegram.
- `analyst` - просмотр dashboard, отчетов, выгрузок и журнала событий.

Текущая ролевая модель проекта должна быть расширена, а не заменена одномоментно.

## Основные сущности

### CRM

- `CrmLead` - нормализованная копия лида из Bitrix24.
- `CrmContact` - контактные данные человека или компании.
- `CrmCompany` - компания, если она есть в CRM.
- `CrmStage` - стадия воронки с внешним кодом Bitrix24.
- `CrmActivity` - дело, звонок, письмо, встреча, задача, ожидание.
- `CrmTimelineEvent` - история событий по лиду.
- `CrmAssignment` - назначение лида или активности на сотрудника.
- `CrmRawPayload` - сырой JSON из Bitrix24 для диагностики и доразбора полей.

### Сотрудники и Telegram

- `ExternalWorker` - внешний сотрудник, его ФИО, статус, график, часовой пояс.
- `MessengerAccount` - связь сотрудника с Telegram chat id/user id.
- `WorkerDailyPlan` - утренний набор задач на день.
- `WorkerDailyReport` - вечерний отчет сотрудника.
- `WorkerReportItem` - ответ по конкретному лиду или активности.
- `ReminderRule` - правила отправки напоминаний.
- `MessageDelivery` - очередь и статус отправки Telegram-сообщений.
- `BotConversationState` - состояние сценария опроса в Telegram.

### Интеграции

- `IntegrationConnection` - настройки подключения к Bitrix24 и Telegram.
- `SyncJob` - запуск синхронизации: ручной, плановый, webhook.
- `IntegrationEvent` - технический журнал входящих/исходящих событий.
- `IdempotencyKey` - защита от повторной обработки webhooks и повторной записи.
- `OutboundCommand` - команда на запись в Bitrix24 или Telegram.

## Потоки данных

### Bitrix24 -> Наша система

1. Система получает лиды, контакты, ответственных, стадии, дела и историю.
2. Данные сохраняются в нормализованные таблицы.
3. Сырой payload сохраняется отдельно для диагностики.
4. Внешние id Bitrix24 становятся уникальными ключами синхронизации.
5. Повторная загрузка обновляет существующие записи, а не создает дубли.

Минимальный набор данных для первой версии:

- lead id;
- title;
- stage id/name;
- status semantic;
- assigned by id/name;
- contact name;
- phone;
- email;
- source;
- created date;
- updated date;
- next activity date;
- activity subject/type/status;
- comments/history where available.

### Наша система -> Telegram

Утром сотрудник получает:

- список лидов и дел на сегодня;
- просроченные дела;
- лиды без следующего действия;
- краткие кнопки: `В работе`, `Сделано`, `Нужна помощь`, `Перенести`, `Комментарий`.

Вечером сотрудник получает опрос:

- что выполнено;
- что перенесено;
- где нужен следующий шаг;
- какие лиды заблокированы;
- комментарии для руководителя или CRM.

### Telegram -> Наша система

Ответы сотрудника сохраняются как:

- отчет за день;
- событие по лиду;
- изменение статуса внутренней задачи;
- комментарий;
- запрос помощи;
- причина переноса или невыполнения.

### Наша система -> Bitrix24

На первом этапе запись назад должна быть ограниченной и контролируемой:

- добавить комментарий в таймлайн лида;
- создать дело или задачу;
- закрыть дело как выполненное;
- обновить стадию только по разрешенным правилам;
- зафиксировать результат вечернего отчета.

Любая запись назад должна иметь audit trail и idempotency key.

## Аналитика

Новые dashboard-блоки:

- лиды по стадиям;
- новые лиды за период;
- конверсия между стадиями;
- среднее время в стадии;
- лиды без следующего дела;
- просроченные активности;
- нагрузка по сотрудникам;
- выполнение утреннего плана;
- доля вечерних отчетов, отправленных вовремя;
- проблемные лиды и причины блокеров;
- повторные лиды;
- источники лидов и качество обработки.

## Архитектурные решения

Рекомендуемые решения для первой production-like версии:

- оставить Express + Prisma + PostgreSQL;
- добавить отдельные backend modules: `bitrix`, `telegram`, `leadops`, `scheduler`, `integrations`;
- хранить normalized data и raw payload одновременно;
- синхронизацию делать идемпотентной;
- для фоновых задач использовать простой scheduler на старте, затем вынести в очередь;
- Telegram webhook сделать отдельным endpoint;
- Bitrix webhooks и pull-синхронизацию поддержать одновременно;
- не давать Telegram-боту напрямую менять Bitrix24 без записи события в нашу БД.

## Риски

- Bitrix24 API может иметь лимиты и разные форматы полей для лидов, дел и таймлайна.
- Сотрудники могут отвечать неполно или не по сценарию, поэтому нужны кнопки и fallback на свободный текст.
- Запись обратно в Bitrix24 должна быть ограничена правами и журналироваться.
- Сырые CRM payload могут содержать персональные данные, нужна политика хранения.
- Telegram chat id должен привязываться через подтверждение, а не вручную без проверки.
- Нужно заранее решить, кто является источником истины для стадии лида: Bitrix24 или наша система.
