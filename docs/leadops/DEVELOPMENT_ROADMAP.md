# План разработки по итерациям

## Принцип разработки

Разработка должна идти не через большой переписанный релиз, а через последовательные вертикальные срезы:

1. сначала понять текущие данные и API Bitrix24;
2. затем положить CRM-данные в собственную БД;
3. затем подключить Telegram как рабочий интерфейс сотрудника;
4. затем добавить двустороннюю запись;
5. затем довести аналитику и эксплуатационные механизмы.

Каждая итерация должна иметь Scope, Out of scope, Definition of Done и проверочные команды.

## Итерации

### Iteration 00 - Audit and Domain Discovery

Цель: зафиксировать новую предметную область, поля Bitrix24, стадии, роли сотрудников и минимальный рабочий сценарий.

Результат: доменная карта, список CRM-полей, маппинг стадий, словарь терминов, уточненный MVP.

Файл: `iterations/ITERATION_00_AUDIT_AND_DOMAIN.md`.

### Iteration 01 - CRM Data Model

Цель: добавить в Prisma модели лидов, CRM-активностей, сотрудников, Telegram-аккаунтов и интеграционных событий.

Результат: миграция БД, seed для демо-лидов, базовые API чтения лидов.

Файл: `iterations/ITERATION_01_CRM_DATA_MODEL.md`.

### Iteration 02 - Bitrix24 Read Sync

Цель: реализовать загрузку данных из Bitrix24 в нашу систему.

Результат: Bitrix API client, ручной sync endpoint, хранение raw payload, upsert без дублей.

Файл: `iterations/ITERATION_02_BITRIX_SYNC.md`.

### Iteration 03 - Telegram Identity and Bot Skeleton

Цель: подключить Telegram-бота и безопасно связать Telegram chat с внешним сотрудником.

Результат: webhook, команды `/start`, привязка сотрудника, тестовая отправка сообщений.

Файл: `iterations/ITERATION_03_TELEGRAM_IDENTITY_BOT.md`.

### Iteration 04 - Daily Workflows

Цель: реализовать утренние напоминания и вечерние опросы.

Результат: планировщик, шаблоны сообщений, кнопки ответов, сохранение отчетов.

Файл: `iterations/ITERATION_04_DAILY_WORKFLOWS.md`.

### Iteration 05 - Bitrix24 Write-back

Цель: вернуть результаты работы сотрудника из нашей системы в Bitrix24.

Результат: комментарии, дела, закрытие активностей, ограниченное изменение стадий.

Файл: `iterations/ITERATION_05_BITRIX_WRITEBACK.md`.

### Iteration 06 - Lead Analytics

Цель: заменить старую project/finance аналитику на lead operations аналитику.

Результат: dashboard по стадиям, SLA, сотрудникам, отчетам, просрочкам и конверсии.

Файл: `iterations/ITERATION_06_ANALYTICS.md`.

### Iteration 07 - Operations, Security and Demo Readiness

Цель: стабилизировать систему, закрыть эксплуатационные риски и подготовить демо-сценарий.

Результат: инструкции запуска, env example, диагностика интеграций, ручной retry, audit, demo checklist.

Файл: `iterations/ITERATION_07_OPERATIONS_SECURITY.md`.

## MVP

Минимальный полезный продукт:

- ручная загрузка лидов из Bitrix24;
- список лидов и активностей в нашей системе;
- привязка внешнего сотрудника к Telegram;
- утреннее сообщение с задачами;
- вечерний отчет по задачам;
- сохранение ответа в нашей БД;
- dashboard по лидам, просроченным активностям и сотрудникам;
- ручная запись комментария обратно в Bitrix24.

## Что сознательно отложить

- полноценный BI-конструктор;
- сложные роботы Bitrix24;
- автоматическое принятие решений о смене стадии без подтверждения;
- массовую рассылку по нескольким мессенджерам;
- мобильное приложение;
- замену Bitrix24 как основной CRM.

## Порядок проверки после каждой итерации

```bash
cd backend
npm run build
```

```bash
cd frontend
npm run build
```

Дополнительно для интеграционных итераций:

- проверить health endpoint;
- проверить миграции на чистой базе;
- проверить seed;
- проверить один ручной сценарий в UI;
- проверить одну запись в `AuditLog`;
- проверить одну запись в `IntegrationEvent`.
