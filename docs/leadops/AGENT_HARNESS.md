# Agent Harness для LeadOps

Этот файл задает рабочий протокол для Codex/агентов при разработке новой системы. Цель - не дать агенту расползтись за пределы итерации и сохранить проверяемость результата.

## Общий цикл итерации

Каждая итерация проходит через пять шагов:

1. Context intake - прочитать файл итерации, связанные документы и релевантный код.
2. Plan - кратко описать цель, scope, файлы изменений, риски и проверки.
3. Implementation - менять только файлы, необходимые для scope.
4. Verification - выполнить команды проверки и ручной smoke test.
5. Report - перечислить сделанное, проверки, ограничения и следующий шаг.

## Запрос на планирование

```text
Прочитай docs/leadops/iterations/ITERATION_XX_NAME.md.

Сначала не пиши код. Подготовь план:
1. кратко перескажи цель итерации;
2. перечисли текущие модули проекта, которые затрагиваются;
3. предложи структуру реализации по шагам;
4. укажи файлы, которые будут созданы или изменены;
5. укажи риски и спорные места;
6. сопоставь каждый пункт Definition of Done с конкретной проверкой.

После этого остановись и дождись подтверждения.
```

## Запрос на реализацию

```text
План принимаю. Реализуй итерацию строго по docs/leadops/iterations/ITERATION_XX_NAME.md.

Важно:
- не выходи за Scope;
- не делай пункты из Out of scope;
- не переписывай старые модули без необходимости;
- сохраняй совместимость существующего запуска backend/frontend;
- обнови документацию, если меняются команды запуска, env или структура данных.

После реализации:
1. проверь Definition of Done;
2. выполни доступные команды проверки;
3. напиши, что сделано;
4. напиши, что не сделано;
5. укажи, как вручную проверить результат.
```

## Запрос на исправление ошибки

```text
При проверке возникла ошибка:

[вставить ошибку]

Исправь только причину этой ошибки. Не добавляй новую функциональность и не меняй архитектуру без необходимости.
```

## Запрос на ревью

```text
Проведи ревью текущей итерации.

Проверь:
- соответствие Scope;
- невыполненные пункты Definition of Done;
- миграции и совместимость БД;
- безопасность интеграций;
- idempotency внешних событий;
- обработку ошибок Bitrix24 и Telegram;
- тестовые и ручные проверки.

Не пиши новую функциональность.
```

## Правила для интеграций

- Любое внешнее событие сначала фиксируется в нашей БД, затем влияет на доменные сущности.
- Для Bitrix24 и Telegram хранится raw payload.
- Повторная обработка webhook не должна создавать дубли.
- Ошибка внешнего API не должна терять событие.
- Запись обратно в Bitrix24 должна иметь audit trail.
- Секреты не коммитятся и хранятся только в `.env`.

## Формат отчета агента

В конце работы агент пишет:

- `Сделано` - короткий список фактических изменений;
- `Проверено` - команды и результаты;
- `Ограничения` - что осталось вне итерации;
- `Следующий шаг` - какой файл итерации брать дальше.
