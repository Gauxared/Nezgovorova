# LeadOps CRM + Telegram

Этот раздел фиксирует новый вектор проекта: использовать существующий каркас аналитической системы как основу для управления внешними сотрудниками, которые обрабатывают лиды компании.

Целевая связка:

```text
Bitrix24 CRM <-> Наша система <-> Telegram
```

## Документы

- `CURRENT_STATE_ANALYSIS.md` - что проект уже умеет и где границы текущего каркаса.
- `TARGET_SYSTEM_DESIGN.md` - целевая предметная область, сущности, потоки данных и архитектура.
- `DEVELOPMENT_ROADMAP.md` - план полноценной разработки по итерациям.
- `AGENT_HARNESS.md` - правила работы для Codex/агентов по полу-harness методике.
- `iterations/` - отдельные итерационные задания с Scope, Out of scope, Definition of Done и проверками.

## Ключевой вывод

Текущий проект уже полезен как технический каркас: есть Express API, Prisma/PostgreSQL, React frontend, роли, CRUD, аудит, dashboard, отчеты и импорт CSV. Но предметная модель сейчас описывает клиентов, проекты, задачи и финансы. Для новой задачи ее нужно переориентировать на лиды, CRM-активности, сотрудников, Telegram-коммуникации, синхронизацию и аналитику воронки.

## Рабочая гипотеза

Система должна стать операционным слоем между Bitrix24 и внешними исполнителями:

- Bitrix24 остается первичным CRM-источником по лидам, контактам, стадиям, делам и истории.
- Наша система хранит нормализованную копию CRM-данных, историю синхронизаций, правила напоминаний, ответы сотрудников и расчеты аналитики.
- Telegram становится рабочим интерфейсом внешнего сотрудника: утренний список задач, вечерний отчет, быстрые кнопки статусов, комментарии, причины блокеров.
- Из Telegram данные возвращаются в нашу систему и при необходимости записываются обратно в Bitrix24 как дела, комментарии, изменения статуса или новые задачи.
