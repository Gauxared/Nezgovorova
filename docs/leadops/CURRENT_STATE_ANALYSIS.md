# Анализ текущего состояния проекта

## Назначение текущего каркаса

Проект сейчас представляет собой fullstack-приложение для аналитической отчетности. Он был сделан вокруг сущностей `Client`, `Project`, `Task`, `FinancialValue`, справочников, отчетов и dashboard.

Текущий стек:

- backend: Node.js, TypeScript, Express, Prisma, PostgreSQL, JWT, bcrypt, multer;
- frontend: React, TypeScript, Vite, React Router, Recharts;
- database: PostgreSQL;
- интеграции: отсутствуют, кроме CSV-импорта;
- фоновые задачи, очереди, Telegram bot, Bitrix24 API: отсутствуют.

## Backend

Backend расположен в `backend/`.

Реализовано:

- `GET /api/health` с проверкой подключения к БД;
- JWT-аутентификация: `POST /api/auth/login`, `GET /api/auth/me`;
- role-based доступ через `requireRoles`;
- CRUD для пользователей, клиентов, проектов, задач, финансовых записей;
- справочники статусов, направлений и отчетных периодов;
- dashboard analytics на backend;
- отчеты с сохранением фильтров и CSV-экспортом;
- CSV-импорт клиентов, проектов, задач и финансовых значений;
- audit log для create/update/delete/import/export/login.

Текущие ограничения backend:

- нет Bitrix24 клиента, webhooks, idempotency-слоя и журнала интеграционных событий;
- нет модели лида как отдельной CRM-сущности;
- нет модели внешнего сотрудника, Telegram-аккаунта и рабочих смен;
- нет фонового планировщика для утренних и вечерних сообщений;
- нет очереди отправки сообщений и retry-политики;
- нет webhook endpoint для Telegram;
- нет записи обратно в Bitrix24;
- текущая аналитика считает проекты, задачи и финансы, а не лиды, стадии и коммуникации.

## Database

Основная Prisma-схема:

- `Role`;
- `User`;
- `Client`;
- `Direction`;
- `Status`;
- `Project`;
- `Task`;
- `ReportPeriod`;
- `FinancialValue`;
- `Report`;
- `AuditLog`.

Для новой задачи эти таблицы можно использовать только частично:

- `User` можно расширить до внутренних пользователей и операторов системы;
- `Task` может стать базой для внутренних задач, но CRM-дела лучше хранить отдельно;
- `Status` можно переиспользовать как справочник, но нужны CRM-стадии с внешними кодами Bitrix24;
- `AuditLog` полезен, но нужен отдельный `IntegrationEvent` для технической трассировки обмена.

Ключевой разрыв: нет сущностей `Lead`, `LeadStage`, `CrmActivity`, `ExternalWorker`, `MessengerAccount`, `DailyCheckIn`, `DailyReport`, `SyncJob`, `IntegrationEvent`.

## Frontend

Frontend расположен в `frontend/`.

Реализовано:

- login;
- sidebar layout;
- dashboard;
- CRUD-страницы для клиентов, проектов, задач, финансов;
- отчеты;
- импорт CSV;
- audit log;
- справочники.

Текущие ограничения frontend:

- интерфейс отражает старую предметную область;
- нет lead list, lead card, funnel board, stage aging, SLA и истории коммуникаций по лиду;
- нет экрана настройки Bitrix24 интеграции;
- нет экрана настройки Telegram-бота и привязки сотрудников;
- нет рабочего кабинета внешнего сотрудника;
- нет панели качества обработки лидов.

## Что видно на скриншотах Bitrix24

Папка `Screen/` содержит два скриншота CRM.

Из списка лидов видны поля:

- лид;
- стадия;
- дела;
- полное имя;
- дата создания;
- ответственный;
- путь клиента;
- причина проигрыша.

Из карточки лида видны данные:

- текущая стадия;
- сумма и валюта;
- контакт: ФИО, телефон, email;
- источник;
- ответственный;
- комментарий;
- UTM-метки;
- товары;
- история событий;
- входящее письмо;
- быстрые действия: дело, комментарий, сообщение, задача, письмо, ожидание, встреча, звонок.

Видимые стадии воронки:

- Новая/Поиск контакта;
- Отправлено письмо-знакомство;
- Встреча Проведена;
- Переговоры;
- Обсуждение пилота;
- Presale;
- Завершить обработку лида.

Важное следствие: новая система должна учитывать не только статусы лидов, но и дела/активности вокруг лида, потому что именно они становятся основой для утренних напоминаний и вечерних отчетов.

## Текущий технический статус запуска

Проект был поднят локально на штатных портах:

- backend: `http://127.0.0.1:4000`;
- frontend: `http://127.0.0.1:5173`;
- PostgreSQL: `127.0.0.1:5432`;
- health-check: `GET /api/health`.

В `backend/.env` сейчас используется `127.0.0.1` вместо `localhost`, чтобы избежать проблем Windows/IPv6 при подключении к PostgreSQL.

## Вывод

Проект годится как каркас, потому что уже есть авторизация, REST API, БД, Prisma, frontend layout, audit и аналитические страницы. Но его нельзя считать почти готовой CRM-интеграцией: основная работа впереди и будет состоять из новой доменной модели, интеграционного слоя Bitrix24, Telegram-бота, планировщика, двусторонней синхронизации и переработанной аналитики.
